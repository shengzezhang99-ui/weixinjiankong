## 3. 今日排查过的问题

### 3.1 最初发现的问题

打包版进行 OCR 测试时，界面提示没有识别到文字，或者提示缺少 OCR 依赖。

源码环境下 OCR 正常，说明问题更可能集中在打包产物运行环境，而不是规则匹配或截图逻辑本身。

### 3.2 已做过的修复尝试

已把 PaddleOCR 模型缓存显式加入 PyInstaller 打包数据：

```text
C:\Users\Administrator\.paddleocr\whl
```

打包后模型会进入：

```text
dist\微信强提醒助手\_internal\paddleocr_models
```

已确认包内包含 9 个模型文件，包括：

- det 模型
- rec 模型
- cls 模型
- `inference.pdmodel`
- `inference.pdiparams`
- `inference.pdiparams.info`

### 3.3 中文路径问题

排查时发现 Paddle 的底层推理库无法稳定打开中文路径下的模型文件。

已增加运行时处理：

- 如果内置模型路径包含中文，先复制到 ASCII 缓存目录。
- 默认目标目录：

```text
C:\Users\Administrator\AppData\Local\WechatAlertAssistant\paddleocr_models
```

然后让 PaddleOCR 使用缓存目录下的模型。

在源码模拟冻结路径测试中，这个方案可以正常识别已有截图文字。

## 4. 当前未解决问题

用户实际运行打包好的程序时，仍然提示缺少 OCR 依赖。

这说明当前修复还没有完全覆盖真实 exe 运行时环境。可能原因包括：

1. 打包版运行时缺少 PaddleOCR 动态导入的某个隐藏模块。
2. Flet 子进程启动后工作目录或资源路径与模拟测试不一致。
3. exe 内部导入 `paddleocr` 时失败，但源码模拟测试没有覆盖到同样路径。
4. Paddle / PaddleOCR 在 PyInstaller 环境下还有额外 DLL 或 data 文件没有被正确收集。
5. 依赖检测逻辑只显示“缺少 OCR 依赖”，实际异常信息没有完整透出到界面。

目前应按“打包版真实运行日志”为准继续查。

## 5. 明天建议排查顺序

1. 先运行打包版，复现“缺少 OCR 依赖”提示。
2. 查看打包目录下的日志：

```text
D:\weixinjiankong\dist\微信强提醒助手\logs\app.log
```

3. 在 `_refresh_dependency_status` 和 `OcrEngine.available` 里把真实异常写入日志或界面，不只显示“缺少 OCR 依赖”。
4. 用调试版 spec 打一个带控制台窗口的版本，直接看 import traceback。
5. 重点确认以下导入在 exe 内是否成功：

```python
import paddle
import paddleocr
from paddleocr import PaddleOCR
```

6. 如果确认是隐藏依赖问题，再按 traceback 补 `hiddenimports` 或 `collect_all`。
7. 如果确认是 DLL / 模型路径问题，再调整 PyInstaller data/binary 收集或运行时复制目录。
