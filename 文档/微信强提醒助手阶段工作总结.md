# 微信强提醒助手阶段工作总结

> 历史阶段记录。当前实现和后续任务请以 `微信强提醒助手工作交接摘要_2026-07-21.md` 为准。

更新时间：2026-07-15

## 1. 当前目标

本项目目标是做一个 Windows 桌面端微信强提醒助手：

1. 监控指定微信群或聊天窗口。
2. 发现 `@我`、指定名称或关键字后触发强提醒。
3. 强提醒包含置顶弹窗、循环提示音、处理确认。
4. 用户超时未处理时，通过模拟点击通知其他联系人。

当前已经从最初的“右下角微信弹窗 OCR”思路，调整为更现实的多方案验证：

1. 固定区域 OCR 监控。
2. UI Automation 控件树读取验证。
3. 微信窗口定位 + 自动截图 OCR，作为下一阶段主方向。

## 2. 已完成内容

### 2.1 Flet 桌面界面

```powershell
.\run_wechat_alert_py311.ps1
```

当前界面能力：

3. 区域 OCR 监控页。
4. UIA 控件监控实验页。

3. 内容区域支持滚动，低分辨率下不会遮挡底部内容。
4. 强提醒弹窗按钮栏保持可见。
5. 确认类弹窗尽量置顶显示。

### 2.2 强提醒流程

当前强提醒流程：

1. 监控命中规则。
2. 播放报警音。
3. 弹出强提醒窗口。
4. 用户点击“我已处理”后，会出现二次确认。
5. 点击“已确认”后：
   - 关闭提醒弹窗。
   - 停止报警音。
   - 停止当前监控。
   - 更新主界面通知状态。

已修复的问题：

1. 点击测试通知后主界面无法操作。
2. 需要确认的弹窗不在最上层。
3. 提醒弹窗触发后底部三个操作按钮消失。
4. 点击确认后弹窗没有立即消失。

### 2.3 报警音

当前支持常见音频格式：

1. `mp3`
2. `ogg`
3. `wav`
4. `flac`

实现方式：

1. 优先使用 `pygame` 播放配置的音频文件。
2. 未配置或播放失败时，回退到系统提示音。

### 2.4 通知方式

当前只保留模拟点击通知。

模拟点击通知逻辑：

1. 用户先打开联系人聊天窗口。
2. 在界面中记录输入框鼠标位置。
3. 触发通知时程序点击该位置。
4. 粘贴通知模板。
5. 可选按 Enter 发送。

注意：

该方式依赖屏幕坐标。如果联系人窗口移动、被遮挡或输入框位置变化，可能点错。

## 3. OCR 监控现状

### 3.1 右下角弹窗 OCR

最初方案是监控微信右下角新消息弹窗，然后截图 OCR。

该链路目前判断为不适合作为主方案，原因：

1. 右下角弹窗通常只能知道哪个群来了消息。
2. 很多情况下拿不到完整消息内容。
3. 无法可靠判断 `@我` 或关键字。
4. 群免打扰、系统通知设置、微信弹窗策略都会影响触发。

结论：

右下角弹窗 OCR 可以保留为辅助或历史功能，但不适合作为主链路。

### 3.2 固定区域 OCR

当前可用的稳定链路是固定区域 OCR：

1. 用户把目标微信聊天窗口独立出来。
2. 在软件中框选聊天消息区域。
3. 程序定时截图该区域。
4. PaddleOCR 识别文字。
5. 根据 `@名称` 和关键字触发提醒。

优点：

1. 不依赖微信控件树。
2. 对微信 4.x Qt/C++ 自绘窗口也能工作。
3. 实现风险低。

缺点：

1. 依赖窗口位置稳定。
2. 受 OCR 准确率影响。
3. 多群同时监控能力有限。

当前判断：

固定区域 OCR 应继续作为兜底方案保留。

## 4. UI Automation 验证结论

### 4.1 本阶段新增内容

已经新增 UIA 实验链路：

1. 新增配置节 `uia`。
2. 新增模块 `wechat_alert_assistant/uia_monitor.py`。
3. Flet 界面新增 `UIA监控` 页面。
4. 支持 `UIA 自检`。
5. 支持读取一次消息列表。
6. 支持启用 UIA 轮询监控。

依赖：

```text
uiautomation>=2.0.20
```

该依赖已经加入：

```text
requirements-real-monitor.txt
```

### 4.2 实现思路

UIA 读取器当前逻辑：

1. 通过 Windows UI Automation 枚举顶层窗口。
2. 找到疑似微信窗口。
3. 在微信窗口控件树中查找名称包含 `消息` 的 List 控件。
4. 读取 List 子项的 `Name` 文本。
5. 将读取到的文本包装成内部 `Message`。
6. 交给现有规则引擎判断是否触发提醒。

为了避免启动时把历史消息全部触发，UIA 监控默认：

1. 第一次读取作为初始快照。
2. 只记录已见消息。
3. 后续新增消息才参与规则判断。

### 4.3 实际验证结果

在当前机器上自检结果为：

```text
已找到微信窗口，但没有找到“消息”列表控件；当前微信版本可能没有暴露 UIA 消息树
窗口 Name=微信 ClassName=Qt51514QWindowIcon
未发现 List 类型控件
```

该结果说明：

1. `uiautomation` 可以找到微信顶层窗口。
2. 当前微信窗口类名是 `Qt51514QWindowIcon`。
3. 微信内部聊天区没有暴露标准 UIA List 控件。
4. 不能通过 UIA 稳定读取聊天消息内容。

### 4.4 UIA 结论

对于微信 4.x Qt/C++ 自绘窗口：

1. UIA 通常只能读到顶层窗口信息。
2. 大概率读不到聊天消息列表。
3. 也读不到单条消息文本、发送人、`@我` 和关键字内容。

只有当 UISpy / Inspect 能看到类似以下结构时，UIA 才可行：

```text
List: 消息
  ListItem: 张三：@你 处理一下
```

如果 UISpy / Inspect 只能看到：

```text
Window: 微信
ClassName: Qt51514QWindowIcon
```

则 UIA 读取消息内容不可行。

当前结论：

UIA 不适合作为微信 4.x 的主监控链路，但可以保留为实验性自检工具。

## 5. 不建议继续投入的路线

### 5.1 右下角弹窗监控

不建议继续作为主方向。

原因：

1. 信息不完整。
2. 不可靠。
3. 无法准确判断关键字和 `@我`。

### 5.2 UIA 直接读消息

对微信 4.x 不建议继续作为主方向。

原因：

1. 微信 4.x 使用 Qt/C++ 自绘窗口。
2. 消息区域不暴露标准 UIA 控件树。
3. 只能拿到顶层窗口，不能拿到消息内容。

### 5.3 数据库解析

不建议作为当前项目主方案。

原因：

1. PC 微信消息数据库通常涉及加密。
2. 可能需要密钥获取、内存读取或逆向。
3. 版本兼容成本高。
4. 安全和合规风险较高。

### 5.4 Hook / 注入 / 内存读取

不建议走。

原因：

1. 稳定性差。
2. 维护成本高。
3. 容易触发安全风险。
4. 对账号安全不友好。

## 6. 下一阶段建议

建议下一阶段将主方向调整为：

```text
微信窗口定位 + 自动聊天区截图 OCR
```

也就是：

1. 不再指望 UIA 读取消息文本。
2. 只用 Win32/UIA 找到微信窗口位置。
3. 根据窗口矩形自动推算聊天消息区域。
4. 自动截图该区域。
5. 送入 OCR。
6. 做文本去重。
7. 根据 `@名称` 和关键字触发强提醒。

相比当前手动框选区域 OCR，该方案的优势：

1. 用户不用每次手动框选。
2. 微信窗口移动后可以重新定位。
3. 能保留 OCR 对自绘窗口的兼容性。
4. 技术风险比数据库解析、Hook、注入低很多。

建议保留的模式：

1. 自动窗口 OCR：下一阶段主模式。
2. 固定区域 OCR：兜底模式。
3. UIA 自检：实验工具，默认关闭。
4. 弹窗 OCR：保留但弱化，不作为主链路。

## 10. 阶段结论

当前最重要的结论：

```text
微信 4.x 下，UIA 直接读取消息内容大概率不可行。
```

当前最现实的技术路线：

```text
自动定位微信窗口 + 截图 OCR + 规则匹配 + 强提醒
```

固定区域 OCR 已经能作为可用原型继续保留。UIA 代码已经完成第一版验证，但更适合作为诊断工具，而不是正式主链路。
