## 1. 本轮目标

本轮工作从阅读历史开发文档和阶段交接材料开始，主要目标是：

1. 确认当前真正可行的微信监控技术路线。
2. 修复 OCR 文本轻微变化造成的重复报警。
3. 避免自动窗口和固定区域对同一消息重复提醒、重复通知。
4. 补齐可重复执行的测试和安全重启能力。
5. 修复监控设置页的布局问题，并将技术参数改为普通用户可以理解的描述。
6. 整理项目文档、配置示例和版本控制基础设施。

## 2. 当前技术路线

当前正式主链路为：

```text
定位微信聊天窗口
    -> 截取聊天消息区域
    -> PaddleOCR 识别
    -> 文本清洗和去重
    -> 群名、@名称、关键字匹配
    -> 本地强提醒
    -> 超时通知联系人
```

功能定位：

- 自动窗口监控：主模式。
- 固定区域监控：自动窗口识别不稳定时的兜底模式。
- 微信弹窗监控：历史兼容功能，新配置默认关闭。
- UIA：已退出正式功能，不再投入开发。

## 3. OCR 去重重构

### 3.1 原有问题

原代码使用精确文本 hash 判断重复，存在以下问题：

- 同一个名称被识别为 `SI`、`Sl` 等轻微变体时，会被当成新消息。
- 自动窗口和固定区域共用一个去重字典，但使用不同过期时间清理。
- 自动窗口较短的去重时间可能提前删除固定区域应继续保留的记录。
- 屏幕上持续显示的旧消息在去重时间到期后可能再次报警。
- 两种监控模式同时看到同一条消息时，可能分别创建 Alert。

### 3.2 已完成实现

新增 `wechat_alert_assistant/dedup.py`，实现：

- 每个 OCR 来源独立维护已见文本。
- OCR 行按相似度判重，不再要求字符串完全一致。
- 同一次截图中的重复行只保留一次。
- 仍在当前截图中显示的文本会刷新保留时间。
- 文本离开截图并超过保留时间后，才允许作为新消息再次出现。
- 自动窗口和固定区域在短时间内识别到同一规则证据时，只创建一次提醒。
- 跨来源合并只处理不同监控来源，不会仅凭普通文本相似就吞掉未配置消息。

新增配置：

| 配置项 | 默认值 | 实际含义 |
|---|---:|---|
| `dedup_similarity` | `0.88` | 两行 OCR 文本达到多高相似度才视为同一行 |
| `cross_source_dedup_seconds` | `12` | 两种监控识别到同一消息时的提醒合并时间 |
| `fuzzy_name_threshold` | `0.88` | OCR 名称与配置的 @ 名称达到多高相似度才算命中 |

## 4. 规则匹配改进

`wechat_alert_assistant/rules.py` 已将 @ 名称从精确包含升级为模糊匹配。

已验证的典型情况：

```text
配置名称：@SI Example
OCR 结果：@Sl Example
结果：可以识别为 @我
```

关键字仍优先采用精确匹配，避免短关键字模糊匹配产生过多误报。

## 5. Alert 与通知状态修正

本轮补充了以下状态保护：

- 同一个 Alert 的通知线程仍使用 `notify_inflight` 防止重复执行。
- 用户确认处理时，会清理当前活动 Alert。
- Alert 已确认处理后，不再接受迟到的异步通知结果改写状态。
- 已处理 Alert 再次请求通知时会直接忽略。
- 通知失败后会按配置自动重试，默认再试 2 次、间隔 60 秒。
- 达到重试上限后保留失败状态，不再继续自动通知；用户仍可手动点击“立即通知”再试。

模拟点击通知仍属于真实外部操作，会控制鼠标、粘贴文本，并可能按 Enter 发送。本轮自动测试没有调用真实通知。

## 6. 自动窗口裁剪提示

自动窗口 OCR 的识别质量高度依赖消息区范围。

本轮增加了裁剪面积检查：

- 如果选择区域覆盖微信窗口面积超过 85%，测试结果会提示裁剪范围过大。
- 界面明确建议使用“框选消息区”，只保留消息列表。
- 裁剪参数改为“从左、上、右、下分别裁掉多少像素”，不再只写抽象的“边距”。

## 7. 监控设置界面调整

### 7.1 灰色遮挡问题

问题表现：监控设置页出现一整块灰色区域，看起来像遮罩。

根因：Flet 的换行 Row 设置了 `wrap=True`，其子控件同时使用 `expand=True`，Flutter 无法完成该布局，最终显示灰色错误占位。

处理结果：

- 移除了窗口选择行冲突的 `wrap=True`。
- 固定区域页换行布局中的扩展容器改为稳定宽度。
- 后台抓取实际窗口画面确认灰色区域已经消失。

### 7.2 参数文案

原来的“轮询间隔、最近行数、去重秒数、行相似阈值、跨模式去重”等术语已替换。

当前界面使用：

| 当前文案 | 用户实际控制的结果 |
|---|---|
| 识别间隔（秒） | 每隔多久读取一次屏幕 |
| 一次检查聊天末尾多少行 | 每轮只处理聊天区末尾多少行 |
| 同一消息多久内不再提醒（秒） | 重复消息保持静默的时间 |
| 两行多像才算重复（0-1） | OCR 有少量错字时是否仍视为同一行 |
| 两种监控重复提醒合并（秒） | 自动窗口和固定区域看到同一消息时只提醒一次 |
| 名字多像才算 @ 我（0-1） | @名称允许多少 OCR 错字 |

每个复杂参数下方已增加推荐值，以及调高、调低的影响说明。

## 8. 重启脚本

新增：

```text
restart_wechat_alert_py311.ps1
```

使用方式：

```powershell
.\restart_wechat_alert_py311.ps1
```

预览操作但不真正重启：

```powershell
.\restart_wechat_alert_py311.ps1 -WhatIf
```

脚本行为：

1. 只匹配本项目虚拟环境启动的 `-m wechat_alert_assistant` 根进程。
2. 自动查找该进程创建的 Python、Flet 等子进程。
3. 先尝试正常关闭窗口。
4. 超时后只强制结束仍未退出的目标进程。
5. 使用项目 Python 3.11 虚拟环境重新启动一个实例。
6. 不会结束微信、其他 Python 程序或其他桌面应用。

本轮曾发现两套助手 Python 实例同时存在。执行重启脚本后已收敛为单实例。

## 13. 遗留问题

1. 当前配置的报警音文件实际无效，程序会回退到系统提示音，需要更换有效音频。
2. 自动窗口和固定区域的裁剪范围仍需结合真实微信字体、窗口大小和 DPI 现场调整。
3. `0.88` 是当前经验默认值，需要通过实际误报、漏报日志继续校准。
4. 模拟点击通知仍依赖固定屏幕坐标，窗口移动后可能点错。
5. 通知失败自动重试已补基础策略，但仍需要用隔离的模拟点击窗口验证。
6. 托盘、PyInstaller 打包、多显示器和高 DPI 尚未完成系统验收。
7. 尚未建立 Git 初始提交和正式版本标签。
